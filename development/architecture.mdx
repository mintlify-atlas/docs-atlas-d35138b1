## Monorepo overview

RedEye is organized as a Yarn workspaces monorepo managed by [Moon](https://moonrepo.dev/) (`@moonrepo/cli` v1.9.2). Moon handles cross-package task orchestration — builds, linting, formatting, and dev servers — through per-package `moon.yml` task files and a shared `tasks.yml` at the root `.moon/` directory.

The workspace root `package.json` declares four workspace glob patterns:

```
applications/**   — runnable applications
packages/**       — shared library packages
parsers/**        — C2 log parser binaries
docs/*            — documentation site
```

## Directory structure

```
RedEye/
├── applications/
│   ├── client/          # React frontend
│   ├── server/          # Node.js backend + GraphQL API
│   └── redeye-e2e/      # Cypress end-to-end tests
├── packages/
│   ├── graph/           # Graph visualization library
│   ├── models/          # Shared TypeORM data models
│   ├── migrations/      # Database migration scripts
│   └── ui-styles/       # Shared Blueprint.js styles and components
├── parsers/
│   ├── cobalt-strike-parser/
│   ├── brute-ratel-parser/
│   ├── sliver-parser/
│   └── parser-core/     # Shared parser utilities
├── .moon/               # Moon build system configuration
└── scripts/             # Release and utility scripts
```

## Parsers

Parsers are standalone Node.js binaries that read raw C2 log files and write structured data into the SQLite database. The server spawns them as child processes when a campaign is parsed.

| Package | C2 Framework |
|---|---|
| `@redeye/cobalt-strike-parser` | Cobalt Strike |
| `@redeye/brute-ratel-parser` | Brute Ratel C4 |
| `@redeye/sliver-parser` | Sliver |
| `@redeye/parser-core` | Shared parsing utilities |

All three parser binaries are declared in the root `package.json` `bin` field and bundled alongside the server binary in release builds.

## Data flow

```
C2 Log Files
    │
    ▼
Parser binary (child process)
    │  writes entities
    ▼
SQLite database (MikroORM)
    │
    ▼
Apollo Server / TypeGraphQL
    │  /graphql endpoint
    ▼
React client (graphql-request)
    │
    ▼
@redeye/graph  ──▶  SVG canvas in browser
```

1. The user uploads a campaign (raw C2 logs or a `.redeye` file) through the client.
2. The server saves uploaded files to disk and spawns the appropriate parser binary.
3. The parser reads the log files and persists structured entities into SQLite via MikroORM.
4. Once parsing is complete, the client queries data through the GraphQL API.
5. The `@redeye/graph` package renders the beacon-host-server graph in an SVG canvas and responds to timeline scrubbing events.
